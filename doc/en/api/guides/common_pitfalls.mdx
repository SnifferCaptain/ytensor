# Common Pitfalls

This guide lists common issues and solutions when using YTensor.

## 1. Forgetting Shared Memory in Views

**Issue**: Modifying a view unexpectedly affects the original tensor.

```cpp
auto tensor = YTensor<float, 2>::ones(10, 10);
auto sub = tensor.slice(0, 0, 5);  // First 5 rows

sub.fill(0);  // The first 5 rows of the original tensor also become 0!
```

**Solution**: Use `clone()` when an independent copy is needed.

```cpp
auto sub = tensor.slice(0, 0, 5).clone();
sub.fill(0);  // Does not affect the original tensor
```

---

## 2. Using view() on Non-Contiguous Tensors

**Issue**: Calling `view()` on a non-contiguous tensor throws an exception.

```cpp
auto t = tensor.transpose();
auto v = t.view({new_shape});  // ❌ Throws exception!
```

**Solution**: Use `reshape()` or call `contiguous()` first.

```cpp
// Method 1: Use reshape (automatically handles it)
auto v = t.reshape({new_shape});

// Method 2: Make contiguous first
auto v = t.contiguous().view({new_shape});
```

---

## 3. Type Mismatch in Data Access

**Issue**: Accessing data using the wrong template type.

```cpp
YTensorBase t = YTensorBase::zeros({10}, "int32");
float* ptr = t.data<float>();  // ❌ Type reinterpretation error!
```

**Solution**: Check `dtype` before access.

```cpp
if (t.dtype() == "int32") {
    int32_t* ptr = t.data<int32_t>();
    // ...
} else if (t.dtype() == "float32") {
    float* ptr = t.data<float>();
    // ...
}
```

---

## 4. In-Place Operations Affecting Shared Views

**Issue**: In-place operations modify unexpected tensors.

```cpp
auto a = YTensor<float, 2>::ones(3, 4);
auto b = a;  // Shallow copy, shares memory

a += 1.0f;   // b also changes!
```

**Solution**: Understand the memory model and use `clone()` when necessary.

```cpp
auto a = YTensor<float, 2>::ones(3, 4);
auto b = a.clone();  // Deep copy

a += 1.0f;   // b is unaffected
```

---

## 5. Different Slice Step Rules from Python

**Issue**: Reverse slicing results are not as expected.

```cpp
// Python: arr[5:2:-1] returns [5, 4, 3]
// YTensor: start=5, end=2, step=-1 might behave differently
auto result = vec.slice(0, 5, 2, -1);
```

**Solution**: Use `autoFix=true` or understand the rules.

```cpp
// autoFix automatically handles cases where start > end
auto result = vec.slice(0, 5, 2, -1, true);
```

---

## 6. Unchecked Division by Zero

**Issue**: Division results in inf or nan.

```cpp
auto result = tensor / 0.0f;  // Produces inf
auto mod = int_tensor % 0;     // ❌ Undefined behavior (may crash)
```

**Solution**: Check the denominator yourself.

```cpp
if (denominator != 0) {
    result = tensor / denominator;
}
```

---

## 7. Out of Bounds Access with atData_()

**Issue**: `atData_()` does not perform boundary checks.

```cpp
auto tensor = YTensor<float, 1>::zeros(10);
tensor.atData_(100) = 1.0f;  // ❌ Out of bounds! Undefined behavior
```

**Solution**: Ensure the index is within valid range.

```cpp
if (index >= 0 && index < tensor.size()) {
    tensor.atData_(index) = 1.0f;
}
```

---

## 8. Using atData_() After Forgetting Contiguous()

**Issue**: Using `atData_()` on a non-contiguous tensor yields incorrect results.

```cpp
auto t = tensor.transpose();
// t is non-contiguous, atData_() accesses by physical index, giving wrong logical results
float val = t.atData_(0);
```

**Solution**: Make contiguous first or use `atData()`.

```cpp
// Method 1: Use after contiguous
t.contiguous_();
float val = t.atData_(0);

// Method 2: Use atData() (automatic coordinate conversion, slower)
float val = t.atData(0);
```

---

## 9. Bitwise Operations on Floating Point Types

**Issue**: Generally, `%` is considered not to support floating point numbers.

**Fact**: YTensor extends support for `%` using `fabs` and `fmod` internally.

**Note**: Although supported, be aware of floating point precision issues.

```cpp
auto t = YTensorBase::zeros({10}, "float32");
auto result = t % 2.0f;  // ✅ Correct
```

- [ ] Is `atData_()` index valid?
- [ ] Are integer/float operators correct?
